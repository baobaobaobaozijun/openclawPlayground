### 2.3 异常处理流程

```
发现异常
  ↓
判断异常类型
  ├─ 状态文件 > 2 小时未更新 → 询问状态，提醒更新
  ├─ 任务进度滞后 > 4 小时 → 了解原因，协调资源
  ├─ 问题未上报但已阻塞 → 批评教育，强调上报机制
  ├─ 依赖关系未通知 → 立即通知，补充流程
  └─ 工作日志缺失 → 要求补录，记录违规
  ↓
记录到 MEMORY.md
  ↓
必要时启动紧急站会
```

### 2.4 检查工具

#### 状态检查脚本
```powershell
# 检查各 Agent 状态文件更新时间
Get-ChildItem -Path "F:\openclaw\agent\status" -Filter "*.md" | 
  Select-Object Name, LastWriteTime | 
  ForEach-Object { 
    $age = (Get-Date) - $_.LastWriteTime
    if ($age.TotalHours -gt 2) {
      Write-Host "$($_.Name) 超过 2 小时未更新！" -ForegroundColor Red
    } else {
      Write-Host "$($_.Name) 正常 ({$([int]$age.TotalMinutes)} 分钟前)" -ForegroundColor Green
    }
  }
```

#### 工作日志检查脚本
```powershell
# 检查各 Agent 今日工作日志
$agents = @("guantang", "jiangrou", "dousha", "suancai")
foreach ($agent in $agents) {
  $logs = Get-ChildItem -Path "F:\openclaw\agent\workinglog\$agent" -Filter "*.md" |
    Where-Object { $_.LastWriteTime.Date -eq (Get-Date).Date }
  Write-Host "$agent : $($logs.Count) 条日志"
}
```

## 📅 第三部分：计划执行保障

### 3.1 任务分解原则

#### SMART 原则
- **S (Specific):** 任务必须具体明确，不能模糊
- **M (Measurable):** 必须有明确的完成标准
- **A (Achievable):** 任务难度与 Agent 能力匹配
- **R (Relevant):** 任务与项目目标相关
- **T (Time-bound):** 必须有明确的截止时间

#### 任务分解示例

❌ **错误示例:**
```
任务：开发用户认证模块
时间：尽快
```

✅ **正确示例:**
```
任务 ID: TASK-003
任务名称：用户认证模块
负责人：酱肉
开始时间：2026-03-13 09:00
截止时间：2026-03-14 18:00
验收标准:
  - [ ] 手机号登录 API 完成
  - [ ] JWT Token 生成和验证完成
  - [ ] 单元测试覆盖率 ≥ 80%
  - [ ] Swagger 文档完整
依赖关系: TASK-001 (数据库设计) 必须完成
输出文件: AuthService.java, JwtTokenProvider.java, AuthController.java
```

### 3.2 任务分配流程

```
PM 创建任务
  ↓
明确任务信息（ID、名称、负责人、时间、验收标准）
  ↓
识别依赖关系（前置任务、后置任务）
  ↓
sessions_spawn 通知负责人
  ↓
负责人 10 分钟内确认
  ↓ 未确认
PM 追问确认
  ↓
记录到进度仪表板
  ↓
开始执行
```

### 3.3 任务执行监控

#### 监控维度
| 维度 | 监控方式 | 频率 |
|------|----------|------|
| 进度 | 状态文件进度百分比 | 每 2 小时 |
| 质量 | 工作日志、Git 提交 | 每日 |
| 风险 | 问题登记表、站会汇报 | 实时 + 每日 |
| 依赖 | 依赖关系图、完成通知 | 实时 |

#### 进度偏差处理
| 偏差程度 | 处理方式 |
|----------|----------|
| 滞后 < 2 小时 | 提醒，无需升级 |
| 滞后 2-4 小时 | 了解原因，记录风险 |
| 滞后 4-8 小时 | 协调资源，调整计划 |
| 滞后 > 8 小时 | 紧急站会，重新规划 |

### 3.4 顺序执行保障

#### 依赖关系管理

**依赖登记表:**
```markdown
| 任务 ID | 任务名称 | 负责人 | 前置任务 | 后置任务 | 状态 |
|---------|----------|--------|----------|----------|------|
| TASK-001 | 数据库设计 | 酸菜 | - | TASK-002 | ✅ 完成 |
| TASK-002 | 后端初始化 | 酱肉 | TASK-001 | TASK-003 | ✅ 完成 |
| TASK-003 | 用户认证 | 酱肉 | TASK-002 | TASK-014 | 🟡 进行中 |
| TASK-014 | 登录页面 | 豆沙 | TASK-003 | - | ⏳ 等待中 |
```

#### 依赖通知流程
```
前置任务完成
  ↓
负责人更新状态文件（标记完成）
  ↓
负责人 sessions_spawn 通知后置任务负责人
  ↓
后置任务负责人确认收到
  ↓
PM 更新进度仪表板
  ↓
后置任务可以开始
```

#### 阻塞预防
- PM 每日检查依赖关系表
- 前置任务截止前 4 小时提醒
- 前置任务完成后 10 分钟内必须通知
- 后置任务等待超过 1 小时立即上报

### 4.4 资源保障措施

#### 技术资源保障
- **文档先行:** 每个任务开始前，PM 确保相关文档已就绪
- **模板提供:** PM 提供任务模板、代码模板、日志模板
- **知识共享:** 建立统一知识库 `F:\openclaw\agent\doc\`

#### 环境资源保障
- **环境检查:** 酸菜每日 09:00 检查开发/测试环境状态
- **故障响应:** 环境故障 30 分钟内响应，2 小时内修复
- **备份机制:** 关键数据每日备份

#### 时间资源保障
- **合理工期:** PM 分配任务时预留 20% 缓冲时间
- **优先级管理:** 高优先级任务优先保障时间
- **避免多任务:** 同一 Agent 同时进行任务 ≤ 2 个

#### 人力资源保障
- **技能匹配:** 任务分配考虑 Agent 专长
- **跨 Agent 协作:** 复杂任务可多 Agent 协作
- **专家支持:** 技术难题可申请外部专家支持
