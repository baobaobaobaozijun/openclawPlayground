# Agent 任务分发模板 v2.0

> 核心原则：PM 管"要什么"和"怎么验收"，不管"怎么写代码"。
> Agent 保持技术自主权，但必须遵守交付标准。

---

## 通用交付规范（所有 spawn 必须包含）

```
**交付规范（必须遵守）：**
1. 所有代码/文档必须用 write 工具写入文件系统，仅在对话中回复代码 = 未完成
2. 完成后用 exec 执行 Test-Path 验证文件存在
3. 记录工作日志到 workinglog/{agent}/
4. 执行 git add + git commit（在 F:\openclaw\agent 或对应 code 目录）
5. 回复 PM：文件路径 + 验证结果 + commit hash
```

## 豆沙（前端）模板

**特点：** 能写代码但不落盘。必须强调 write 工具。

```
【PM 灌汤 — 任务分配】{任务名称}

**背景：** {上下文}

**目标：** {具体要交付什么}

**技术要求：**
- {具体技术规格}

**交付物：**
- 文件路径：`{期望的文件路径}`

**⚠️ 重要 — 交付规范：**
1. **必须用 write 工具将代码写入文件系统**
2. **仅在对话中回复代码 ≠ 完成，必须落盘**
3. 写入后用 exec 执行 `Test-Path '{文件路径}'` 验证
4. 工作日志 → workinglog/dousha/
5. git add + commit
6. 回复：文件路径 + Test-Path 结果 + commit hash

**如果遇到 write 工具问题：**
- 尝试先创建目录：`New-Item -ItemType Directory -Force -Path '{目录}'`
- 如仍失败，回复具体错误信息，PM 协助

**截止：** {时间}
```

## 酸菜（运维）模板

**特点：** 能执行单命令，但多步骤流程会停滞。要求逐步执行并记录。

```
【PM 灌汤 — 任务分配】{任务名称}

**背景：** {上下文}

**目标：** {具体要交付什么}

**执行步骤：**（按顺序逐步执行，每步执行后记录结果）

| 步骤 | 命令/操作 | 预期结果 |
|------|----------|---------|
| 1 | {具体命令} | {预期输出} |
| 2 | {具体命令} | {预期输出} |
| ... | ... | ... |

**⚠️ 重要 — 执行纪律：**
1. **每执行一个步骤，立即记录实际输出**（不能创建空模板）
2. 遇到失败的步骤，记录错误信息后继续下一步（不要停）
3. 全部完成后汇总结果写入工作日志
4. 工作日志 → workinglog/suancai/
5. git add + commit
6. 回复：每步结果 + 失败步骤的错误信息

**截止：** {时间}
```
