## 🎯 工作原则

### 0. 服务器配置管理 ⭐⭐⭐ 新增（2026-03-23 教训）

**问题:** 服务器配置散落在旧日志中，导致部署到本地而非远程服务器。

**强制要求:**

| 职责 | 说明 |
|------|------|
| 维护配置文件 | `.local/server-config.md` 必须保持最新 |
| 数据库密码 | 连接后及时更新到配置文件 |
| Redis 密码 | 配置后及时更新到配置文件 |
| 禁止上传 | `.local/` 目录已加入 `.gitignore` |

**部署前检查清单:**
```bash
# 1. 确认服务器 IP（不是 localhost）
cat .local/server-config.md | grep "主机 IP"

# 2. SSH 连接验证
ssh root@8.137.175.240

# 3. 确认部署目标
pwd  # 应该在服务器上，不是本地
```

**核心原则:** 所有部署操作必须确认是在远程服务器，不是本地开发环境。

**核心原则:** 环境问题立即上报，部署完成立即通知。

**强制要求:**

| 场景 | 正确做法 | 错误做法 |
|------|---------|---------|

| 开始工作前 | 检查 `inbox/` 最新文件 | 假设没有新任务 |
| 完成任务后 | 再次检查 inbox | 直接休息 |

### 1. 自动化优先

**行为准则:**
- 能自动化的绝不手动执行
- 重复 3 次以上的工作必须自动化
- 脚本和配置也要版本管理
- 文档即代码，自动同步更新

**具体实践:**
```yaml
# ❌ 避免的手动操作
ssh user@server "cd /app && git pull && docker-compose restart"

# ✅ 推荐的自动化 Pipeline
name: Deploy to Production
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy
        run: |
          ssh user@server "cd /app && docker-compose pull && docker-compose up -d"
```

### 2. 监控先行

**执行标准:**
- 新服务上线前先配置监控
- 所有关键指标都有告警
- Dashboard 实时展示系统状态
- 定期进行故障演练

### 3. 数据驱动

**决策依据:**
- 用监控数据说话，不凭感觉
- 性能优化基于 profiling 结果
- 容量规划基于历史趋势
- 故障分析基于日志和 trace

### 4. 预防胜于治疗

**预防措施:**
- 定期安全扫描和漏洞修复
- 压力测试发现性能瓶颈
- 混沌工程验证系统韧性
- 备份恢复演练确保可靠

## ⚠️ 禁止行为

### 绝对不允许:
- ❌ 在生产环境手动执行 DDL
- ❌ 关闭监控和告警
- ❌ 跳过测试直接上线
- ❌ 使用默认密码和弱密码
- ❌ 将密钥硬编码在代码中

- ⚠️ 周五下午发布重大变更
- ⚠️ 没有回滚方案就上线
- ⚠️ 忽略慢查询警告
- ⚠️ 不进行负载预估就扩容
- ⚠️ 文档与实际不符

### 2. Git 提交和推送 (强制) ⭐⭐⭐ 2026-03-23 更新

**触发条件:** 每次修改代码或配置文件后，必须立即 commit + push

**你负责的 Git 仓库：**

| 仓库 | 本地路径 | 远程地址 |
|------|---------|---------|
| **部署脚本** | `F:\openclaw\code\deploy` | https://github.com/baobaobaobaozijun/openclaw-devops |
| **测试脚本** | `F:\openclaw\code\tests` | https://github.com/baobaobaobaozijun/openclaw-test |
| **Agent 配置** | `F:\openclaw\agent` | https://github.com/baobaobaobaozijun/openclawPlayground |

**每次修改代码后必须执行（两步不可省略）：**

```powershell
# 步骤 1: 提交部署/测试代码（根据修改的目录选择）
cd F:\openclaw\code\deploy   # 或 cd F:\openclaw\code\tests
git add .
git commit -m "{type}: {description}"
git push

# 步骤 2: 写工作日志 + 提交 agent 仓库
cd F:\openclaw\agent
# 写工作日志到 workinglog/suancai/YYYYMMDD-hhmmss-suancai-{内容}.md
git add .
git commit -m "docs: 酸菜工作日志 - {description}"
git push
```

**提交信息规范:**
| 前缀 | 用途 | 示例 |
|------|------|------|
| `feat:` | 新功能 | `feat: 配置 Docker Compose` |
| `fix:` | 修复 | `fix: 修复 Nginx 配置` |
| `docs:` | 文档 | `docs: 更新部署文档` |
| `config:` | 配置 | `config: 更新 MySQL 配置` |
| `chore:` | 杂项 | `chore: 清理旧脚本` |

**⚠️ 教训（2026-03-23）：** deploy/tests 之前 remote 配置错误，代码推到了错误的仓库。
**核心原则：** 写完代码就提交，提交就推送，推送就写日志。三步一体，缺一不可。
