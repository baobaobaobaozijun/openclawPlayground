## ❤️ 心跳监控机制 (PM 职责)

**触发方式:** Cron 定时任务（每 10 分钟自动执行） + 自动唤醒机制 ⭐  
**配置文件:** `.openclaw/crons/agent-monitoring.yml`  
**机制文档:** [doc/guides/agent-heartbeat-mechanism.md](../doc/guides/agent-heartbeat-mechanism.md)

**核心职责:**
1. **主动检查** - 通过 Cron 每 10 分钟自动检查各 Agent 状态
2. **识别失联** - 发现最后活动 > 60 分钟的 Agent
3. **🆕 自动唤醒** - 发现失联 Agent 立即创建 inbox 通知文件唤醒
4. **分配任务** - 为空闲 Agent 立即分配新任务
5. **维护看板** - 更新 `doc/progress/agent-heartbeat-dashboard.md`
6. **协调协作** - 确保所有 Agent 都有明确任务并持续工作

## 🆕 自动唤醒机制 ⭐⭐⭐ (2026-03-17 新增)

### 唤醒触发条件

**满足任一条件即触发唤醒:**
- ❌ 工作日志 > 60 分钟无更新
- ❌ 代码目录 > 120 分钟无修改
- ❌ 未响应站会任务分配（>30 分钟）
- ❌ 未执行心跳汇报（>2 小时）

### 唤醒流程

```
心跳检查（每 10 分钟）
  ↓
检查各 Agent 最后活动时间
  ↓
发现失联 Agent（>60 分钟无活动）
  ↓
创建 inbox 紧急通知文件
  ↓
Git 提交并推送
  ↓
等待响应（10 分钟）
  ↓
有回复 → 分配任务 → 持续监控
  ↓
无回复 → 记录违规 → 升级处理
```

### 唤醒通知模板

**文件路径:** `workspace-{agent}/inbox/emergency-notify-{YYYYMMDD}-{HHmm}.md`

**通知内容:**
```markdown
# 紧急通知 - 立即响应 🔴

**发送者:** 灌汤 (PM)  
**发送时间:** {时间}  
**优先级:** CRITICAL  
**要求响应时间:** 10 分钟内

## 🚨 失联警报

{Agent} 你好，PM 心跳检查发现你已失联超过 X 小时！

**最后活动:** {时间}  
**失联时长:** X 小时  
**今日任务:** {任务 ID}（{截止时间} 截止）

## ⏰ 立即汇报要求

请**立即**阅读站会纪要并开始工作：
**站会纪要:** `doc/meetings/daily-standup-{YYYYMMDD}-0900.md`

**汇报内容:**
1. 是否收到站会纪要？
2. 今日是否开始工作？当前进度（%）？
3. 接下来 1 小时计划？
4. 能否按时交付？

**汇报方式:**
1. 更新 `MEMORY.md` 添加当前状态
2. 创建今日工作日志
3. 开始执行任务

## ⚠️ 逾期处理

如 10 分钟内仍未响应，PM 将：
1. 记录严重违规到绩效考核
2. 调整任务优先级和交付时间
3. 考虑人工介入检查 Gateway 状态
```

### 唤醒后跟进

| 时间 | 行动 | 说明 |
|------|------|------|
| 0 分钟 | 创建 inbox 通知 | 紧急唤醒文件 |
| 2 分钟 | Git 提交推送 | 确保文件同步 |
| 10 分钟 | 检查响应 | 读取工作日志/MEMORY.md |
| 15 分钟 | 有回复→分配任务 | 开始正常工作流程 |
| 15 分钟 | 无回复→记录违规 | 更新 MEMORY.md 违规记录 |
| 30 分钟 | 二次唤醒 | 升级通知，警告人工介入 |
| 60 分钟 | 人工介入 | 检查 Gateway/重启服务 |

## 🔍 详细监控流程

### 每次 Cron 触发时（每 10 分钟）

**必须执行:**

#### 1. 检查各 Agent 状态

**酱肉 (后端) 🍖:**
- ✅ workinglog/jiangrou/ 今日日志数量
- ✅ workspace-jiangrou/code/backend/ 最后修改时间
- ✅ workspace-jiangrou/MEMORY.md 最后更新
- ✅ 判断：正常/空闲/失联

**豆沙 (前端) 🍡:**
- ✅ workinglog/dousha/ 今日日志数量
- ✅ workspace-dousha/code/frontend/ 最后修改时间
- ✅ workspace-dousha/MEMORY.md 最后更新
- ✅ 判断：正常/空闲/失联

**酸菜 (运维) 🥬:**
- ✅ workinglog/suancai/ 今日日志数量
- ✅ workspace-suancai/ 最后活动时间
- ✅ workspace-suancai/MEMORY.md 最后更新
- ✅ 判断：正常/空闲/失联

#### 2. 判定标准

**🟢 正常:**
- 今日有工作日志
- 最后活动 < 60 分钟
- 有明确任务在进行

**🟡 空闲:**
- 今日有工作日志
- 最后活动 30-60 分钟
- 任务已完成或无明确任务

**🔴 失联:**
- ❌ 今日无工作日志
- ❌ 最后活动 > 60 分钟
- ❌ 未响应通知/任务

#### 3. 采取行动

**🟢 正常 →** 继续观察，更新看板

**🟡 空闲 →** 
1. 检查 tasks/inbox/是否有待处理任务
2. 如无任务，从项目计划分配新任务
3. 创建任务文件到 inbox
4. 通知 Agent 开始执行

**🔴 失联 →** 
1. 🆕 **立即创建 inbox 紧急通知**
2. 🆕 **Git 提交并推送**
3. 等待响应 10 分钟
4. 有回复 → 分配任务
5. 无回复 → 记录违规 → 二次唤醒

#### 4. 更新心跳看板

- 文件：`doc/progress/agent-heartbeat-dashboard.md`
- 记录每个 Agent 的最后活动时间
- 标记当前任务和状态
- 记录本次检查采取的行动
- 🆕 记录唤醒次数（如有）

### 失联判定标准

**满足任一条件即判定为失联:**
- ❌ 最后活动时间 > 60 分钟（workinglog 无新日志）
- ❌ 代码目录 > 120 分钟无修改
- ❌ subagent 唤醒后 10 分钟无回复
- ❌ 对话历史 > 120 分钟无发言

**失联唤醒流程:**
```
发现失联 Agent
  ↓
使用 sessions_spawn 直接调用
  ↓
要求 10 分钟内汇报：
1. 当前任务及进度
2. 最后活动时间及内容
3. 是否有技术阻碍
4. 预计下次汇报时间
  ↓
等待回复
  ↓
有回复 → 检查工作日志 → 分配任务
  ↓
无回复 → 第二次唤醒（紧急）→ 记录风险 → 人工介入
```

**subagent 唤醒模板:**
```
【心跳检查 - 立即汇报】🔴

灌汤 PM 心跳检查发现你已失联超过 X 小时。

请立即汇报：
1. 当前任务及进度
2. 最后活动时间及内容
3. 是否有技术阻碍
4. 预计下次汇报时间

⏰ 限时 10 分钟内回复
```

### 空闲判定标准

**满足任一条件即判定为空闲:**
- ❌ tasks/inbox/ 为空（无待处理任务）
- ❌ logs/ 显示超过 20 分钟无新活动
- ❌ code/ 目录超过 20 分钟无修改
- ❌ 对话历史中超过 20 分钟无发言

**行动:**
- ✅ 立即在对话中询问：@Agent 请报告当前状态
- ✅ 如无响应，从项目计划分配新任务
- ✅ 如确认空闲，5 分钟内完成新任务分配

### 任务连续性原则

**确保工作流不断档:**
- ✅ 任务 A 完成前，准备好任务 B
- ✅ 发现空闲，立即分配新任务
- ✅ 遇到阻塞，协助切换任务，不要等待

**优先级:**
1. 高优先级：阻塞问题（立即处理）
2. 中优先级：空闲 Agent（5 分钟内分配）
3. 低优先级：常规监控（10 分钟检查一次）

## 📋 强制提交要求

### 文档提交
**触发条件:** 每次 F:\openclaw 文件夹下的文档内容发生更改

**必须执行:**
1. 检查工作区状态 (git status)
2. 添加所有更改的文件 (git add .)
3. 提交更改 (git commit -m "docs: {描述}")
4. 推送到 GitHub (git push)

### 代码提交
**触发条件:** 每次 F:\openclaw 文件夹下的代码内容发生更改

**必须执行:**
1. 检查工作区状态 (git status)
2. 添加所有更改的文件 (git add .)
3. 提交更改 (git commit -m "{type}: {描述}")
4. 拉取最新代码 (git pull --rebase)
5. 推送到 GitHub (git push)

**提交信息规范:**
| 前缀 | 用途 | 示例 |
