# 执行日志记录器

## 触发时机

每当完成用户的更改命令后（包括代码修改、文件创建/删除、配置更新等），必须执行本流程生成工作文档。

## 文档命名规则

文档名称格式：`{yyyyMMdd-HHmmss}-资本家的vscode-{工作内容}.md`

示例：
- `skill-create-execution-log_20260309-143022.md`
- `docker-config-update_20260309-150145.md`
- `code-refactor-authentication_20260309-161230.md`

### 工作内容提取规则

1. **简洁性**：控制在 3-5 个单词或中文词组
2. **描述性**：清晰反映核心工作内容
3. **格式**：使用小写字母，连字符分隔（英文）或使用中文
4. **类型标识**：
   - 文件操作：`file-create`, `file-modify`, `file-delete`
   - 代码操作：`code-add`, `code-modify`, `code-delete`, `refactor`
   - 配置操作：`config-update`, `config-add`
   - 部署操作：`deploy-setup`, `deploy-config`
   - 文档操作：`doc-write`, `doc-update`
   - 技能创建：`skill-create`

## 文档存储位置

**统一存储路径**：`agent/workinglog/`

完整文件名：`agent/workinglog/{工作内容}_{yyyyMMdd-HHmmss}.md`

## 文档结构

```markdown
# {工作内容} 执行记录

## 基本信息
- **执行时间**: {开始时间} - {结束时间}
- **触发命令**: {用户的原始请求}
- **Git Push 序列号**: {自动生成的序号，如 #001, #002}
- **涉及文件**: 
  - 文件 1
  - 文件 2

## 执行内容

### 任务概述
{简要描述执行了什么任务}

### 详细变更

#### 1. {变更类别 1}
{具体变更内容}

#### 2. {变更类别 2}
{具体变更内容}

## 关键决策
{记录执行过程中的关键决策点及原因}

## 遇到的问题
{如有遇到问题，记录下来}

## 后续建议
{可选：对后续工作的建议}

## 附件
{相关代码片段、截图等}
```

## 执行流程

### 步骤 1: 识别变更范围

在执行用户命令前，先分析：
- 将涉及哪些文件
- 预计需要多长时间
- 是否有关键风险点

### 步骤 2: 执行变更

按照标准流程执行用户的更改命令，同时记录：
- 开始时间
- 每个关键步骤的执行情况
- 遇到的问题和解决方案

### 步骤 3: 生成文档

变更完成后立即生成文档：

1. **确定工作内容标签**
   - 从执行的变更中提取核心关键词
   - 转换为符合命名规则的格式

2. **生成时间戳**
   - 使用当前系统时间
   - 格式：`yyyyMMdd-HHmmss`

3. **获取 Git Push 序列号**
   - 读取 `agent/workinglog/.push-sequence.json` 文件
   - 如果没有该文件，初始化为 0
   - 序列号 +1
   - 更新 `.push-sequence.json` 文件
   - 格式化为 `#001`, `#002` 等

4. **创建文档**
   - 存储位置：`agent/workinglog/`
   - 使用标准模板填充内容
   - 确保信息完整准确，包含 Git Push 序列号

### 步骤 4: 确认与反馈

向用户报告：
```
✅ 任务完成

📝 已生成执行文档：{文档文件名}
📍 存储位置：agent/workinglog/{文档文件名}
⏱️ 执行耗时：{X 分钟}
📄 涉及文件：{文件数量} 个
```

## 特殊场景处理

### 场景 1: 小型修改
对于简单的单文件修改，可以简化文档内容，但必须包含：
- 修改时间
- 修改内容摘要
- 涉及文件

### 场景 2: 复杂多步骤任务
对于包含多个子任务的复杂请求：
- 为每个子任务单独记录
- 生成主文档汇总整体进展
- 使用编号区分：`{task}_part1_{timestamp}.md`

### 场景 3: 失败的任务
如果任务执行失败或部分失败：
- 仍然生成文档
- 明确标注失败原因
- 记录已完成的步骤
- 提供恢复建议

## 质量检查清单

生成文档前确认：
- [ ] 文档名称符合格式规范
- [ ] 时间戳准确反映执行时间
- [ ] Git Push 序列号已生成并记录
- [ ] 涉及文件列表完整
- [ ] 变更内容描述清晰
- [ ] 关键决策有合理解释
- [ ] 存储位置正确：`agent/workinglog/`

## 基本信息

- **涉及文件**: 
  - .lingma/skills/execution-log/SKILL.md

## 执行内容

### 任务概述
创建了名为"execution-log"的技能，用于自动记录用户命令执行过程并生成带时间戳的工作文档。

### 详细变更

#### 1. 技能文件创建
- 创建目录：`.lingma/skills/execution-log/`
- 创建文件：`SKILL.md`
- 定义触发条件、文档格式、执行流程等规范

- 统一日志存储位置：`agent/workinglog/`

- 采用 `{工作内容}_{时间戳}.md` 格式确保文档可追溯
- 日志统一存放在 `agent/workinglog/` 便于集中管理

## 注意事项

1. **时间准确性**: 确保记录开始和结束时间
2. **内容真实性**: 客观记录执行过程，不夸大不缩小
3. **格式一致性**: 所有文档遵循统一的模板格式
4. **及时生成**: 任务完成后立即生成文档，不拖延
5. **用户友好**: 文档内容便于用户理解和回顾
6. **路径统一**: 所有日志必须存放在 `agent/workinglog/` 目录
