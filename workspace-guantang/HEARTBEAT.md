**⚠️ 机制 v3.0 重要变更：**
- ✅ **状态持久化** - 从文件改为 MySQL 数据库（6 张表）
- ✅ **自动恢复** - 系统重启后自动续做/重做（restore-pipeline.ps1）
- ✅ **心跳监控** - Agent 主动上报 (5min) + PM 被动检查 (10min)
- ✅ **审计追溯** - 所有状态变更写入 pipeline_state_history

3. **通知主会话** — 唤醒后通知主会话分配任务
4. **维护看板** — 更新 `doc/05-progress/agent-heartbeat-dashboard.md`

## 🚀 回调驱动机制 ⭐⭐⭐ (2026-03-25 新增 — 核心改进)

### 原则：subagent 完成 = 立即行动，不等心跳

**触发时机:** 收到 subagent task completion event 时

**处理流程:**
```
收到 subagent 完成回调
  ↓
【步骤 1】Test-Path 验证交付文件是否存在
  ↓
【步骤 2】Test-Path 检查工作日志是否存在（自动化合规检查）
  ↓
日志缺失 → 标记异常，下轮任务强制要求先补录
  ↓
【步骤 3】如同批次还有其他 Agent 未完成 → NO_REPLY 等待
  ↓
【步骤 4】同批次全部完成 → 批量验证 + git commit
  ↓
【步骤 5】立即派发下一轮任务（sessions_spawn × 3）
  ↓
无需等待心跳，零空窗！
```

### 关键规则

1. **同批次等待:** 3 个 Agent 同时派发的任务视为一批，前 2 个完成时 NO_REPLY，第 3 个完成时统一处理
2. **即时派发:** 批次验证完成后，立即准备并派发下一轮任务，不等下一次心跳
3. **心跳仅兜底:** 心跳检查时如果发现 Agent 已完成但未被派发新任务（回调遗漏），则补派

### 预期效果

- **消除空窗期:** 从 20-60min 降到 <2min
- **日产出提升:** 从 10 轮/天提升到 15-20 轮/天
- **PM token 节约:** 心跳检查变轻量（仅失联检测），不再每次都完整扫描

## 🎯 Agent 能力适配策略 ⭐⭐⭐ (2026-03-26 v4.0 数据验证版)

### 核心教训

**不同 Agent 适合不同类型的任务。任务模式必须匹配 Agent 能力。**

| Agent | 擅长 ✅ | 不擅长 ❌ | 最佳任务模式 |
|-------|--------|----------|-------------|
| 🍖 酱肉 | 写新单文件、SQL 脚本、配置文件 | 参考模仿、多文件、read+edit | **直接给完整代码 → write 单文件** |
| 🍡 豆沙 | 页面组件、样式、响应式 | — | 给需求 + 参考样例，可自由发挥 |
| 🥬 酸菜 | 脚本/配置/文档 | 多步骤复杂命令 | **给完整内容 → write 创建** |

### 酱肉专项规则 🍖 v4.0

**背景:** 2026-03-26 数据分析 — 酱肉停滞任务 100% 集中在"参考现有文件→模仿编写"和"多文件协调"类任务。

**任务派发强制规则:**
1. ✅ **直接给完整代码** — 不要让他参考 UserMapper.java 等文件
2. ✅ **每个 spawn 只写 1 个文件** — 多文件任务拆分为多次 spawn
3. ✅ **≤3 步 tool call** — write + compile + log
4. ✅ **关键数据内联** — 表结构/字段直接贴在消息中
5. ❌ **禁止"参考 xxx 写 yyy"** — 失败率 >80%
6. ❌ **禁止 read+edit** — PM 直接执行
7. ✅ **10 分钟无日志 → sessions_spawn 询问**
8. ✅ **60 分钟无交付物 → PM 直接兜底**

**编译命令固定:** `cd F:\openclaw\code\backend; mvn compile -q`

**工单模板示例:**
```

📤 交付物（单文件）：

### 日志自动合规检查 📝

**触发时机:** 每次 subagent 完成回调时

**检查流程:**
```
收到 Agent 完成回调
  ↓
Test-Path 检查对应工作日志文件是否存在
  ↓
存在 → 标记 ✅ 合规
  ↓
不存在 → 标记 ⚠️ 缺失
  ↓

**连续缺失升级:**
- 1 次缺失 → 下轮任务提醒
- 2 次连续 → 下轮任务简化（降低复杂度）
- 3 次连续 → 暂停该 Agent 派发，PM 直接执行其任务

## 🆕 自动唤醒机制 ⭐⭐⭐ (2026-03-17 新增)

### 唤醒触发条件

**满足任一条件即触发唤醒:**
- ❌ 工作日志 > 60 分钟无更新
- ❌ 代码目录 > 120 分钟无修改
- ❌ 未响应站会任务分配（>30 分钟）
- ❌ 未执行心跳汇报（>2 小时）

### 唤醒流程（2026-03-23 更新）

**⚠️ 教训:** 只创建 inbox 文件 = 无效唤醒，Agent 不会主动阅读。

```
心跳检查（每 10 分钟）
  ↓
检查各 Agent 最后活动时间
  ↓
发现失联 Agent（>60 分钟无活动）
  ↓
【步骤 1】创建 inbox 紧急通知文件（记录用途）
  ↓
【步骤 2】立即 sessions_spawn 强制唤醒 ⭐ 关键
  ↓
指定阅读 inbox 文件 + 执行具体任务
  ↓
等待响应（10 分钟）
  ↓
有回复 → 分配任务 → 持续监控
  ↓
无回复 → 第二次 sessions_spawn 唤醒 → 记录违规
```

**sessions_spawn 唤醒模板:**
```
sessions_spawn(agentId, task="【PM 灌汤 — 紧急唤醒 🔴】

{Agent}，心跳检查发现你已失联超过 X 小时。

2. 汇报当前任务及进度
3. 10 分钟内回复

如无响应，将记录严重违规。")
```
无回复 → 记录违规 → 升级处理
```

## 🔍 详细监控流程

### 每次 Cron 触发时（每 10 分钟） — 轻量兜底模式

**注意：Cron 心跳已降级为兜底角色。主力派发由回调驱动完成。**

**心跳仅需做：**
1. 检查各 Agent 最后活动时间（workinglog 最后文件 LastWriteTime）
2. 如有 Agent >60min 无活动且无活跃 subagent → 派发新任务
3. 如有 Agent >60min 但有活跃 subagent → 不干预（正在执行中）
4. 全员正常 → HEARTBEAT_OK

#### 1. 检查各 Agent 状态

**酱肉 (后端) 🍖:**
- ✅ workinglog/jiangrou/ 今日日志数量
- ✅ workspace-jiangrou/code/backend/ 最后修改时间
- ✅ workspace-jiangrou/MEMORY.md 最后更新
- ✅ 判断：正常/空闲/失联

**豆沙 (前端) 🍡:**
- ✅ workinglog/dousha/ 今日日志数量
- ✅ workspace-dousha/code/frontend/ 最后修改时间
- ✅ workspace-dousha/MEMORY.md 最后更新
- ✅ 判断：正常/空闲/失联

**酸菜 (运维) 🥬:**
- ✅ workinglog/suancai/ 今日日志数量
- ✅ workspace-suancai/ 最后活动时间
- ✅ workspace-suancai/MEMORY.md 最后更新
- ✅ 判断：正常/空闲/失联

#### 2. 判定标准

**🟢 正常:**
- 今日有工作日志
- 最后活动 < 60 分钟
- 有明确任务在进行

**🟡 空闲:**
- 今日有工作日志
- 最后活动 30-60 分钟
- 任务已完成或无明确任务

**🔴 失联:**
- ❌ 今日无工作日志
- ❌ 最后活动 > 60 分钟
- ❌ 未响应通知/任务

#### 3. 采取行动

**🟢 正常 →** 继续观察，更新看板

**🟡 空闲 →** 
1. 检查 tasks/inbox/是否有待处理任务
2. 如无任务，从项目计划分配新任务
3. 创建任务文件到 inbox
4. 通知 Agent 开始执行

**🔴 失联 →** 
1. 🆕 **立即创建 inbox 紧急通知**
2. 🆕 **Git 提交并推送**
3. 等待响应 10 分钟
4. 有回复 → 分配任务
5. 无回复 → 记录违规 → 二次唤醒

#### 4. 更新心跳看板

- 文件：`doc/progress/agent-heartbeat-dashboard.md`
- 记录每个 Agent 的最后活动时间
- 标记当前任务和状态
- 记录本次检查采取的行动
- 🆕 记录唤醒次数（如有）

### 失联判定标准

**满足任一条件即判定为失联:**
- ❌ 最后活动时间 > 60 分钟（workinglog 无新日志）
- ❌ 代码目录 > 120 分钟无修改
- ❌ subagent 唤醒后 10 分钟无回复
- ❌ 对话历史 > 120 分钟无发言

**失联唤醒流程:**
```
发现失联 Agent
  ↓
使用 sessions_spawn 直接调用
  ↓
要求 10 分钟内汇报：
1. 当前任务及进度
2. 最后活动时间及内容
3. 是否有技术阻碍
4. 预计下次汇报时间
  ↓
等待回复
  ↓
有回复 → 检查工作日志 → 分配任务
  ↓
无回复 → 第二次唤醒（紧急）→ 记录风险 → 人工介入
```

**subagent 唤醒模板:**
```
【心跳检查 - 立即汇报】🔴

灌汤 PM 心跳检查发现你已失联超过 X 小时。

请立即汇报：
1. 当前任务及进度
2. 最后活动时间及内容
3. 是否有技术阻碍
4. 预计下次汇报时间

⏰ 限时 10 分钟内回复
```

### 空闲判定标准

**满足任一条件即判定为空闲:**
- ❌ tasks/inbox/ 为空（无待处理任务）
- ❌ logs/ 显示超过 20 分钟无新活动
- ❌ code/ 目录超过 20 分钟无修改
- ❌ 对话历史中超过 20 分钟无发言

**行动:**
- ✅ 立即在对话中询问：@Agent 请报告当前状态
- ✅ 如无响应，从项目计划分配新任务
- ✅ 如确认空闲，5 分钟内完成新任务分配

### 任务连续性原则

**确保工作流不断档:**
- ✅ 任务 A 完成前，准备好任务 B
- ✅ 发现空闲，立即分配新任务
- ✅ 遇到阻塞，协助切换任务，不要等待

**优先级:**
1. 高优先级：阻塞问题（立即处理）
2. 中优先级：空闲 Agent（5 分钟内分配）
3. 低优先级：常规监控（10 分钟检查一次）

3. 提交更改 (git commit -m "{type}: {描述}")
4. 拉取最新代码 (git pull --rebase)
5. 推送到 GitHub (git push)

**提交信息规范:**
| 前缀 | 用途 | 示例 |
|------|------|------|
| `feat:` | 新功能 | `feat: 添加 Agent 健康检查` |
| `fix:` | 修复 | `fix: 修正酱肉仓库地址` |
| `docs:` | 文档 | `docs: 更新 README.md` |
| `config:` | 配置 | `config: 更新 Gateway 端口` |
| `chore:` | 杂项 | `chore: 清理旧配置文件` |
