# 工作流程模板

## 完整工作流程

### 阶段1：需求接收与分析

#### 步骤1.1 需求获取
```yaml
操作：监听用户需求输入
输入：
  - 需求描述
  - 优先级
  - 预期周期
输出：
  - 原始需求记录
  - 初步分类
步骤1.2 需求澄清
操作：与用户交互澄清需求
验证点：
  - 需求范围是否清晰
  - 是否包含隐含需求
  - 优先级是否合理
  - 依赖关系是否存在
输出：
  - 澄清记录
  - 修订后的需求
步骤1.3 需求文档生成
操作：生成结构化需求文档
格式：需求文档 (requirement_YYYYMMDD_001.md)
包含：
  - 需求描述
  - 验收标准
  - 技术约束
  - 依赖关系
文件位置：F:\openclaw\workspace\projects\{project_id}\
阶段2：项目设计
 步骤2.1 可行性分析
 操作：评估项目可行性
    分析项：
      - 技术可行性
      - 资源可行性
      - 风险评估
      - 成本评估
    输出：
      - 可行性报告
      - 风险清单
 步骤2.2 项目规划
    操作：制定项目计划
    定义内容：
      - 项目目标
      - 交付物清单
      - 里程碑（Milestone）
      - 时间表
    输出：
      - 项目规划文档 (project_plan_YYYYMMDD_001.md)
      - 时间表
 步骤2.3 架构设计
 操作：与酱肉Agent沟通架构设计
 信息：
   - 技术选型建议
   - 架构约束
   - 接口定义需求
 输出：
   - 架构设计文档
   - 技术规范
 步骤2.4 设计文档生成
   操作：生成完整设计文档
   格式：项目设计文档 (project_design_YYYYMMDD_001.md)
   位置：F:\openclaw\workspace\projects\{project_id}\design\
阶段3：工作拆分
    步骤3.1 工作项分解
    操作：将项目分解为工作项（WBS）
    原则：
      - 每个工作项应独立可执行
      - 工作项粒度控制在2-5天工作量
      - 清晰定义工作项之间的依赖
    分类：
      - 后端任务（交由酱肉）
      - 前端任务（交由豆沙）
      - 运维/测试任务（交由酸菜）
    输出：
      - 工作分解结构 (WBS)
    步骤3.2 任务优先级排序
    操作：为任务排序
    考虑因素：
      - 依赖关系
      - 关键路径
      - 资源可用性
      - 优先级
    输出：
      - 排序后的任务列表
    步骤3.3 生成任务分配表
    操作：生成正式的任务分配表
    格式：task_allocation_YYYYMMDD_001.md
    内容：
      - 任务ID
      - 任务名称
      - 负责Agent
      - 优先级
      - 预期工作量
      - 截止日期
      - 依赖任务
      - 详细描述
    位置：F:\openclaw\workspace\projects\{project_id}\tasks\
阶段4：任务分发
    步骤4.1 后端任务分发
    操作：调用酱肉Agent
    方法：Agent-to-Agent RPC
    内容：
      - 任务ID列表
      - 每个任务的详细说明
      - 技术规范
      - 交付标准
    事件：
      - 记录分发时间
      - 记录确认状态
    步骤4.2 前端任务分发
    操作：调用豆沙Agent
    方法：Agent-to-Agent RPC
    内容：
      - UI/UX设计要求
      - 前端任务列表
      - 交互规范
      - 性能要求
    步骤4.3 运维/测试任务分发
    操作：调用酸菜Agent
    方法：Agent-to-Agent RPC
    内容：
      - 测试计划
      - 部署规范
      - 监控要求
      - 验收标准
    步骤4.4 任务分发确认
    操作：记录分发结果
    记录内容：
      - 分发时间戳
      - 接收方Agent确认
      - 任务开始时间
      - 预期完成时间
    日志：task_distribution_log_YYYYMMDD.log
阶段5：进度跟踪
    步骤5.1 定期进度收集
    频率：日报（建议每日下午4点）
    操作：
      - 调用各Agent获取进度
      - 收集完成任务清单
      - 记录遇到的问题
    格式：daily_progress_report_YYYYMMDD.md
    步骤5.2 进度分析
    操作：分析进度是否按计划进行
    分析项：
      - 完成率
      - 偏差分析
      - 风险识别
      - 资源状态
    输出：
      - 进度分析报告
    步骤5.3 问题协调
    操作：处理发现的问题
    流程：
      1. 识别阻碍
      2. 分析原因
      3. 制定解决方案
      4. 协调相关Agent
      5. 跟踪解决结果
    步骤5.4 进度更新
    操作：更新项目进度数据
    更新内容：
      - 任务状态（未开始/进行中/已完成/已延期）
      - 完成百分比
      - 实际工作量
      - 剩余工作量
    文件位置：F:\openclaw\workspace\projects\{project_id}\progress\
阶段6：交付验证
    步骤6.1 交付物检查
    操作：验收各Agent提交的交付物
    检查项：
      - 代码质量
      - 功能完整性
      - 文档完整性
      - 符合规范
    步骤6.2 测试结果收集
    操作：收集酸菜Agent的测试结果
    内容：
      - 测试覆盖率
      - 缺陷清单
      - 性能指标
    步骤6.3 最终交付报告
    操作：生成最终交付报告
    格式：final_delivery_report_YYYYMMDD_001.md
    内容：
      - 项目总结
      - 交付物清单
      - 质量指标
      - 经验教训
    位置：F:\openclaw\workspace\projects\{project_id}\delivery\
工作流程决策树    
需求接收
    ↓
[需求是否清晰？] → 否 → 澄清需求 → [已澄清？] → 否 → 要求重新描述
    ↓ 是                                         ↓ 是
需求分析
    ↓
[需求是否可行？] → 否 → 评估风险/成本 → [是否继续？] → 否 → 建议暂停
    ↓ 是                                           ↓ 是
项目设计
    ↓
[设计是否完整？] → 否 → 继续设计 → [完整？] → 否 → 建议设计审查
    ↓ 是                                       ↓ 是
工作拆分
    ↓
[任务是否合理？] → 否 → 重新拆分 → [合理？] → 否 → 与团队讨论
    ↓ 是                                      ↓ 是
任务分发
    ↓
[确认分发完成？] → 否 → 重新分发 → [完成？] → 否 → 人工介入
    ↓ 是                                    ↓ 是
进度跟踪
    ↓
[是否按计划进行？] → 否 → 风险预警 → 协调解决 → 更新计划
    ↓ 是
[项目是否完成？] → 否 → 继续跟踪
    ↓ 是
交付验证
    ↓
[是否通过验收？] → 否 → 反馈问题 → Agent修正 → 重新验收
    ↓ 是
项目完成
文件命名规范
所有生成的文件应遵循以下命名规范：

Code
{文档类型}_{日期YYYYMMDD}_{序号}.md

示例：
- requirement_20260307_001.md
- project_plan_20260307_001.md
- project_design_20260307_001.md
- task_allocation_20260307_001.md
- daily_progress_report_20260307.md
- final_delivery_report_20260307_001.md
